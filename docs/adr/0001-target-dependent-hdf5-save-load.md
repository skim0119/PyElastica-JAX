# Target-dependent HDF5 Save/Load without sync

Simulation I/O is a single `save` / `load` surface over rods, blocks, and simulators. File-level attributes are owned by `save`/`load`; payload groups are written by an `Hdf5StateIO` implementation injected per target: blocks implement `write_hdf5_state` / `read_hdf5_state` natively, simulators gain them via the `Hdf5IO` mixin, and host rods use a package-side adapter because upstream `CosseratRod` cannot grow methods. We deliberately do not call `from_device` / `to_device` inside I/O: a **block** (including blocks reached via a **simulator**) is saved from device state with ghosts preserved and with JAX platform + device id recorded; a user-held **rod** is saved from its host arrays as-is and never counts as a simulator resume artifact. Schema levels 0 / 1 / 10 select field groups (collection names); only a block/simulator file at level 10 is Load-safe for resume, and Load checks device metadata by default (optional disable). One Save produces one HDF5 file (multi-block simulators share that file). This replaces the sketched block-checkpoint path as the persistence model rather than keeping a separate “frame vs checkpoint” API.
